# Documentation Philosophy

Documentation is written from experience, not from theory.

Every document should answer:

- Why does this exist?
- How is it used?
- When should it be updated?

If a lesson required time to learn, it is a candidate for documentation.

If it solved a real problem, it belongs in the Playbook.

The goal is not to remember everything.

The goal is to create a system that makes future development faster, more consistent, and more enjoyable.

# Engineering Principles

- Build from experience.
- Prefer reusable solutions.
- Data before opinions.
- Measure twice, cut once.
- Refactor when the evidence supports it.
- Build today's solution, not tomorrow's problem.
- Small improvements compound over time.
- Every lesson learned should reduce future friction.
