# Documentation Philosophy

Documentation is written from experience, not from theory.

Every document should answer:

- Why does this exist?
- How is it used?
- When should it be updated?

If a lesson required time to learn, it is a candidate for documentation.

If it solved a real problem, it belongs in the Playbook.

The goal is not to remember everything.

The goal is to create a system that makes future development faster, more consistent, and more enjoyable.

---

# Engineering Principles

- Build from experience.
- Prefer reusable solutions.
- Data before opinions.
- Measure twice, cut once.
- Refactor when the evidence supports it.
- Build today's solution, not tomorrow's problem.
- Small improvements compound over time.
- Every lesson learned should reduce future friction.

---

# Development Philosophy

## Build Version 1 First

A working Version 1 is more valuable than a perfect design that never gets built.

Finish the first version.

Improve it through experience.

---

## Capture Ideas Without Losing Focus

Not every good idea belongs in the current sprint.

If it doesn't directly improve the current objective:

- Add it to the roadmap.
- Return to the current task.

Progress is made by finishing work, not by constantly changing direction.

---

## Every Document Has a Defined Scope

Useful information does not automatically belong everywhere.

Each document should have one primary responsibility.

Keep documentation focused.

---

## Teach the Pattern First

Document reusable concepts before documenting specific examples.

Patterns transfer between projects.

Examples reinforce understanding.

---

## Documentation is Part of Development

Documentation is not something written after a project is complete.

It is developed alongside the software.

The documentation should evolve as the project evolves.

---

## Engineering Over Memorization

The objective is not to memorize commands, syntax, or workflows.

The objective is to build systems that reduce the need for memorization.

The Playbook exists to capture experience so future development becomes easier.

---

# Continuous Improvement

This document is expected to evolve.

If a new principle changes the way software is built, documented, or maintained, it belongs here.
